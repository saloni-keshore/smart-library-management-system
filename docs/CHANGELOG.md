## 2026-07-22 — Payment workflow audit: centralized receipt numbering, linked Cashbook entries to their payment, fixed unhandled-crash and negative-amount risks

- **Feature:** The Payment workflow end-to-end (Student→Membership→Payment→Receipt→Cashbook→Audit Log→Dashboard→BI→Notifications), covering Admission, Renewal, and standalone pending-fee Collect Payment — audited as a Phase 4 follow-up to the Financial System, Dashboard, and Membership Workflow audits, scoped to correctness/consistency/duplicate-logic elimination, no UI redesign, no new features.
- **Files changed:** `database/payment_queries.py` (new), `database/cashbook_queries.py`, `database/cashbook_categories.py`, `routes/payment.py`, `routes/membership.py`, `templates/students/{index,view}.html`, `templates/payments/{index,collect}.html`, `templates/memberships/{index,create,renew}.html`, `docs/FILE_REFERENCE.md`, `docs/DIAGRAMS.md`, `docs/09_DEPENDENCY_MAP.md`, `docs/DECISIONS.md`, `docs/11_FUTURE_WORK.md`, `docs/WHERE_TO_MODIFY.md`, `docs/TROUBLESHOOTING.md`.
- **Why:** live-DB verification (real `library.db`, one admin's actual Receipt Settings: `receipt_prefix="ABC"`, `next_receipt_number=3000`) found the payment-recording path had three separate, drifted implementations and one silently-broken schema link. (1) `routes/payment.py`'s `collect()`, `routes/membership.py`'s `create()`, and `renew()` each inlined their own `INSERT INTO payments` and their own receipt-number formula — `collect()` used `REC-YYYYMMDD-<membership_id>-<int(new_paid)>`, `create()`/`renew()` used `REC-YYYYMMDD-<membership_id>` with no suffix at all — and none of the three read `receipt_prefix`/`next_receipt_number` (TD-22, logged 2026-07-20, confirmed still open after two further audit cycles). Verified live: this admin's actual receipts were `REC-20260705-0012` etc., never `ABC-03000` as their own Settings page said they should be. (2) `collect()`'s formula truncated a running `float` total with `int()` for its uniqueness suffix — two partial payments on the same membership the same day landing on the same integer (e.g. running totals of `500.4` and `500.8`, both `int()`-truncating to `500`) would collide on `payments.receipt_number`'s `UNIQUE` constraint, and nothing caught the resulting `sqlite3.IntegrityError` — confirmed by reproducing the collision directly against a scratch copy of the live DB, which raised exactly that unhandled error. (3) `cashbook.payment_id` is declared as a real FK in `schema.sql` but `insert_income_entry()`'s actual `INSERT` never included it — every automatic Cashbook row from a payment has always had `payment_id IS NULL`, confirmed via `PRAGMA table_info`/live query. `database/migrate_backfill_cashbook_payments.py` (a real, already-shipped one-off script) proves this in practice: it can only match a `payments` row to its Cashbook entry by `(admin_id, person, amount, entry_date)`, a heuristic two payments sharing all three would defeat — the FK the schema promises was never actually usable for reconciliation. (4) `create()`/`renew()` validated `paid_amount + due_amount > 0` but not each value individually — a form submission with `paid_amount=-500, due_amount=1000` passed that check (total `500 > 0`) and would have stored a negative `paid_amount` on the membership with no payment record to back it, since the payment-insert block only ran `if paid_amount > 0`. (5) The "Collect Payment" button on the Student profile showed for any membership regardless of balance, linking into a form that could only reject the submission after the fact with "cannot exceed pending balance of ₹0". (6) `PAYMENT_METHODS` (the canonical list driving Cashbook's payment-method filter dropdown and its manual-entry validation) was missing `"Card"`, even though Payment Collect, Membership Create, and Membership Renew have all offered "Card" as a `payment_mode` option — a Card payment could be recorded but never filtered to in Cashbook. (7) Verifying "Success messages"/"Error messages" against the live app (not just reading the code) surfaced a 7th, more fundamental defect: every `flash()` call in `routes/payment.py`/`routes/membership.py` — including the new receipt-number success messages and rollback error messages added by this same fix — was being set correctly and then silently discarded, because none of `templates/students/{index,view}.html`, `templates/payments/{index,collect}.html`, or `templates/memberships/{index,create,renew}.html` (nor the shared `layouts/base.html` they all extend) ever called `get_flashed_messages()`. Confirmed via the Flask test client against a scratch copy of the live DB: a POST that the route logic correctly rejected (e.g. `paid_amount=-500`) still rendered a plain 200 form with zero visible indication anything went wrong. This is a pre-existing, wider gap — 11 of the app's ~25 top-level pages do render flashes (Settings sub-pages, Enquiries list, Auth pages), the rest, including several already covered by earlier audits, don't (see TD-31).
- **Database changes:** None — `cashbook.payment_id` and all `library_settings` receipt columns already existed in `schema.sql`/the live DB; this fix only starts populating/reading columns that were already there but unused.
- **UI changes:** Receipt numbers now reflect each admin's configured prefix/starting number (verified: the test admin now gets `ABC-03000`, `ABC-03001`, ... instead of `REC-20260722-...`) instead of ignoring Settings → Receipt Settings entirely. Payment/Membership success flash messages now include the generated receipt number, **and — for the first time — actually render**: the same Flask-test-client verification confirmed `alert-success`/`alert-danger` banners now genuinely appear in the response HTML for collect success, over-pending rejection, and negative-amount rejection, where before this fix none of the seven Payment-workflow templates displayed any flash at all. The "Collect Payment" button on a Student's profile no longer appears once `pending_amount` is `0`. `PAYMENT_METHODS` now includes "Card", so Card transactions are filterable in Cashbook like every other payment mode. A failed payment/membership save (e.g. an unlikely receipt-number collision) now shows a red flash message with nothing saved, instead of Flask's default error page.
- **Future impact:** Added `database/payment_queries.py` (`generate_receipt_number`, `record_payment`) as the single implementation every payment-creating route now shares (ADR-13) — any future payment-creating flow must call it, not add a fourth inline formula. `insert_income_entry()` gained an optional `payment_id` parameter, now always supplied by `record_payment()` (never by `routes/cashbook.py`'s manual entries, correctly). Verified via a scratch copy of the live DB: receipt allocation, the persisted-counter increment, the `payments`↔`cashbook` `payment_id` link, and end-to-end flash rendering all behave correctly under the Flask test client — including a forced `UNIQUE` collision, which rolled back cleanly with zero orphaned rows in either table, and a repeated double-submit, which (as expected, see TD-30) produced two distinct, individually-valid payments rather than an error. TD-22 flipped to **Resolved (partially — `receipt_prefix`/`next_receipt_number` only)**. Opened TD-29 (`auto_increment_receipt` is read but deliberately not honored — no manual-receipt-number UI exists for "off" to mean anything), TD-30 (no idempotency/double-submit protection on any payment-creating form — each would need a real feature addition, not a quick fix, so both are logged rather than fixed in this audit's correctness/consistency scope), and TD-31 (flash messages invisible on ~14 of the app's ~25 top-level pages, a pre-existing gap this audit's own live-app verification surfaced rather than caught by code reading — partially resolved for the 7 Payment-workflow pages listed above, the rest logged for whichever future audit covers those modules).

## 2026-07-21 — Membership workflow audit: fixed stale status display, duplicate active-membership risk, and wired Membership Settings into Create/Renew

- **Feature:** The Membership workflow end-to-end (Enquiry→Student→Membership create/renew/status/pending-fee/receipt/cashbook/audit/dashboard/BI/notification sync) — audited as a Phase 3 follow-up to the same-day Dashboard (previous entry) and Financial System audits, scoped to correctness/consistency/maintainability, no UI redesign, no new features.
- **Files changed:** `database/membership_queries.py` (new), `routes/membership.py`, `routes/dashboard.py`, `routes/membership_distribution.py`, `routes/student.py`, `routes/notification.py`, `templates/memberships/create.html`, `templates/memberships/renew.html`, `docs/FILE_REFERENCE.md`, `docs/09_DEPENDENCY_MAP.md`, `docs/DIAGRAMS.md`, `docs/DECISIONS.md`, `docs/11_FUTURE_WORK.md`, `docs/WHERE_TO_MODIFY.md`, `docs/TROUBLESHOOTING.md`.
- **Why:** widget-by-widget verification against the live app (Flask test client, real DB) found three real defects plus confirmed two already-logged debt items (TD-6, TD-7). (1) `routes/membership.py`'s `index()` (the `/memberships` list) selected the raw `membership_status` column with no date check, unlike every other screen showing membership status (Dashboard, Student profile, Membership Distribution, Notifications) — confirmed against the live DB: membership_id 12 (student 9) has `membership_status='Active'` but `end_date='2026-07-06'` (15 days in the past relative to today, 2026-07-21), and the Memberships table showed it as "Active" while every other page correctly showed "Expired" for the same row. This was TD-6's predicted "the 3 copies could drift" risk materializing as an actual, user-visible inconsistency in a 4th, previously-unaudited copy. (2) `create()` had no guard against a student who already has a live membership — navigating to `/memberships/create/<id>` for such a student (e.g. via back-button or a stale bookmark) would insert a second simultaneously-`Active` membership row instead of being redirected to `renew()`, silently breaking every "one active membership per student" assumption the rest of the app makes. (3) TD-7: `membership_settings` (admin-configured plan fees/days/admission fee) was saved by Settings but never read by `create()`/`renew()` — both routes' JS hardcoded `{Monthly:30, Quarterly:90, "Half-Yearly":180, Yearly:365}` and left fee entirely to manual typing, so changing Settings → Membership Settings had zero effect on what a new/renewed membership actually charged or how long it lasted. Confirmed live: one admin has Monthly configured at ₹1200/30 days with a ₹1200 admission fee — none of that reached the Create/Renew forms before this fix.
- **Database changes:** None — all fixes are query/routing logic and template JS, not schema.
- **UI changes:** The Memberships list (`/memberships`) now shows "Expired" for a membership whose `end_date` has passed even if the underlying column still says `Active` — matching Dashboard/Student profile/Membership Distribution/Notifications, which already did this. Create/Renew's plan dropdown now auto-fills Duration and Paid Amount from this admin's actual Membership Settings (previously a hardcoded 30/90/180/365 with fee always defaulting to 0) — Create additionally adds the configured admission fee on top of the plan fee for a brand-new membership; the amounts remain editable (e.g. for a discount) rather than enforced. Attempting to create a second membership for a student who already has one now redirects to Renew with a warning flash instead of silently creating a duplicate `Active` row.
- **Future impact:** Added `database/membership_queries.py` as the shared home for "effective status" (`EFFECTIVE_STATUS_SQL`/`get_effective_status`), "days until expiry" (`DAYS_LEFT_SQL`), the active-membership guard (`get_active_membership`), the Dashboard/Membership Distribution active-expired count query (`get_membership_counts`), and the Membership Settings→pricing bridge (`get_plan_pricing`/`get_admission_fee`) — any new membership-status/pricing logic should extend this module, not re-inline the SQL. TD-6 flipped to **Resolved** (all four duplicate-status copies found — `routes/dashboard.py`, `routes/student.py`, `routes/membership_distribution.py`, and the newly-found `routes/membership.py` — now share `EFFECTIVE_STATUS_SQL`/`get_effective_status`). TD-7 flipped to **Partially Resolved** — plan duration/fee/admission-fee are wired in; `late_fee_per_day`, `renewal_grace_days`, `auto_expiry`, `allow_early_renewal` still have no consuming code anywhere (no late-fee line item, no grace-window enforcement, no auto-expiry job, no early-renewal block) and would each require new UI/automation surface, which this audit's "no new features" scope excluded. Opened a new TD (TD-28) documenting that Notification Settings' `reminder_7_days`/`reminder_3_days`/`reminder_1_day`/`notify_on_expiry_day`/`notify_after_expiry` toggles are still fully cosmetic — `routes/notification.py`'s `get_notification_summary()` always computes the same today/tomorrow/3-day/expired buckets regardless of what the admin disabled.

## 2026-07-21 — Dashboard integrity audit: fixed dead script, blank chart empty-state, duplicate queries, conflicting loading JS

- **Feature:** Dashboard (`/dashboard`) as a complete product — KPI cards, charts, tables, notification counts, sidebar, loading states — audited as a Phase 2 follow-up to the same-day financial system audit (previous changelog entry), scoped to correctness/consistency/performance/professionalism only, no new features.
- **Files changed:** `templates/components/dashboard_header.html`, `utils/charts.py`, `database/notification_settings_queries.py`, `app.py`, `routes/dashboard.py`, `database/cashbook_queries.py`, `templates/components/revenue_chart.html`, `static/js/dashboard-charts.js`, `docs/FILE_REFERENCE.md`, `docs/06_TEMPLATES_REFERENCE.md`, `docs/07_STATIC_ASSETS.md`, `docs/11_FUTURE_WORK.md`.
- **Why:** widget-by-widget verification against the live app (Flask test client, real DB) found five real defects. (1) `dashboard_header.html` had a `<script>` calling `document.getElementById("dashboardDate").innerHTML = ...` against a `#dashboardDate` element that's been commented out of the markup — a `TypeError` on every single `/dashboard` page load. (2) `utils/charts.py`'s `generate_membership_chart` (the Dashboard's compact membership pie) had no empty-state handling — confirmed by generating it for an admin with zero memberships: a completely blank white PNG, unlike its sibling `generate_membership_distribution_donut` which already shows "No membership data yet". Fixing it surfaced a second bug in *that* existing empty-state text too: `bbox_inches="tight"` crops to matplotlib's default `(0, 1)` axes extent when no explicit limits are set, so the centered-looking `ax.text(0, 0, ...)` call actually rendered pinned in the bottom-left corner of the image — confirmed visually, fixed in both functions. (3) `app.py`'s global `inject_notification_summary` context processor (runs on every authenticated page) and `routes/dashboard.py`'s `dashboard()` view both independently called `get_notification_settings(admin_id)` — confirmed via call-count instrumentation: 2 identical `SELECT`s per `/dashboard` request for the same row. (4) `routes/dashboard.py` ran two near-identical `COUNT(DISTINCT ...)` queries over the same `memberships JOIN students` scan for Active/Expired Memberships, differing only in the `end_date` comparison direction. (5) Two competing, undocumented chart-loading-reveal mechanisms coexisted in `dashboard-charts.js` — a per-element `data-reveal-delay`-driven CSS-class system, and a second `DOMContentLoaded` handler added later that unconditionally revealed every chart at a hard-coded 1200ms regardless of each element's own configured delay (900ms for two of the three chart cards) — confirmed the second one was only load-bearing because `revenue_chart.html`'s `chart-content` div carried a stray `d-none` class its two sibling chart cards didn't have.

- **UI changes:** No console error on Dashboard load anymore. An admin with zero memberships now sees a properly centered "No membership data yet" message on the Dashboard's membership chart instead of a blank white image (and the Membership Distribution page's donut chart, sharing the same fix, is now correctly centered too — it had the same off-center bug already, just not blank). Dashboard's Revenue Overview and Membership Distribution charts now reveal on their originally-configured staggered delays (900ms) instead of a uniform 1200ms. All KPI numbers verified byte-identical to pre-refactor values (Active/Expired Memberships, Today's Collection) — these were performance/consistency refactors, not value changes.
- **Future impact:** `database/cashbook_queries.py` gained `get_today_fee_collection(admin_id)`, completing the fee-revenue family alongside `get_total_fee_revenue`/`get_pending_fees` from the same day's financial audit (ADR-11) — Dashboard's "Today's Collection" now shares that single source of truth instead of being the last inline-SQL financial figure in `dashboard.py`. `database/notification_settings_queries.py` gained `get_notification_settings_cached(admin_id)` (memoized on `flask.g`), now used by both call sites on a `/dashboard` request — verified via instrumentation that this cut 2 identical queries to 1. Opened TD-27 in [11_FUTURE_WORK.md](11_FUTURE_WORK.md) for a redundancy found but deliberately not fixed (Library Highlights duplicating 2 KPI-card values verbatim) since removing/restructuring visible cards was judged out of this audit's correctness/consistency/performance scope, not a defect in the numbers themselves.

## 2026-07-21 — Financial system audit: unified fee-revenue calculations, fixed health-score/ledger/chart bugs

- **Feature:** Cross-cutting financial calculations used by Dashboard, Cashbook, Payments, Memberships (Membership Distribution), and Business Intelligence.
- **Files changed:** `database/cashbook_queries.py`, `database/bi_queries.py`, `routes/dashboard.py`, `routes/membership_distribution.py`, `routes/business_intelligence.py`, `templates/components/bi_revenue_trend_chart.html`, `utils/charts.py`, `docs/FILE_REFERENCE.md`, `docs/DECISIONS.md`.
- **Why:** a full audit of every income/expense/profit/pending-fee/revenue/collection calculation found: (1) "Total Revenue" (Dashboard) was `SUM(payments.amount_paid)` via inline SQL, "Revenue Collected" (Membership Distribution) was the same figure re-derived as a Python `sum()` over already-fetched rows, and "Pending Fees" was duplicated as a third copy of the same `SUM(memberships.pending_amount)` query in `dashboard.py` — three code paths for two numbers, agreeing only by coincidence of being mathematically equivalent restatements. (2) The Business Health Score's fee-collection component divided fee-specific `pending` by `pending + get_total_income(admin_id)` — but `get_total_income()` sums *all* Cashbook income, including non-fee manual categories (Donation, Library Fine, Book Sale, Other Income). Verified against the live `database/library.db`: one admin has a single mistyped manual "Library Fine" entry of ₹100,000,567, which pushed that admin's health-score collection component to ~99.99% "collected" regardless of real fee-collection performance (true fee-only collection rate is 58.9%). (3) `get_cashbook_ledger`'s `source = "Automatic"` filter used `c.source != 'Cashbook Manual Entry'`, which in SQL evaluates to `NULL` (not true) for the 2 pre-`migrate_cashbook_ledger.py` legacy rows that have `source IS NULL` — those rows disappeared from *both* the "Automatic" and "Manual" ledger filters, confirmed against the live DB (13 total rows, only 11 matched either filter before the fix). (4) The Business Intelligence "Revenue Trend" chart plotted Revenue and Expenses but no Profit line, unlike Cashbook's equivalent chart which already does. (5) The Dashboard's Revenue Overview PNG (`utils/charts.py`'s `generate_revenue_chart`) grouped `payments.payment_date` by `strftime('%m', ...)` with no year filter, so payments from different calendar years would silently sum into the same month bar — dormant today (all data is 2026) but a live bug the moment the app has been used across a year boundary.
- **Database changes:** None — all fixes are query/aggregation logic, not schema. (Not a code fix, but flagged during this audit: `cashbook` entry_id 19 for admin_id 1 is a live data-entry error — ₹100,000,567 "Library Fine" — that distorts that admin's Cashbook/BI totals; left untouched pending the admin's confirmation, since correcting/deleting live financial data wasn't requested and isn't a code change.)
- **UI changes:** Dashboard's "Total Revenue" and "Pending Fees" cards, Membership Distribution's "Revenue Collected" and "Pending Payments" cards, and the Business Health Score gauge now all read from the same underlying queries and can no longer numerically drift apart. The Business Intelligence "Revenue Trend" chart gained a third "Profit" line (purple, `#7c3aed`) and its subtitle now reads "revenue vs expenses vs profit". The Cashbook ledger's "Automatic" transaction-source filter now includes legacy pre-migration rows it previously hid. No visible change to the Dashboard Revenue Overview chart today (all existing data is within one calendar year), but it will no longer silently merge revenue from different years once multi-year data exists.
- **Future impact:** Added ADR-11 to [DECISIONS.md](DECISIONS.md) documenting that fee revenue (`get_total_fee_revenue`, Payments-sourced) and Cashbook income (`get_total_income`, all categories) are deliberately distinct metrics that must never be blended — anyone adding a new "money collected" KPI should call `get_total_fee_revenue`/`get_pending_fees`, not re-derive either. No new `TD-N` opened (all newly-found issues were fixed in this session); the ₹100,000,567 test-data outlier is a data-integrity issue, not tracked as technical debt.

## 2026-07-21 — Data & Backup: removed redundant Export Database action

- **Feature:** Settings → Data & Backup
- **Files changed:** `routes/setting.py`, `templates/settings/data_backup.html`, `docs/05_ROUTES_REFERENCE.md`, `docs/06_TEMPLATES_REFERENCE.md`, `docs/10_FEATURE_MODULES.md`, `docs/FILE_REFERENCE.md`, `docs/WHERE_TO_MODIFY.md`, `docs/01_OVERVIEW.md`, `docs/11_FUTURE_WORK.md`, `docs/DECISIONS.md`
- **Why:** `Export Database` (`backup_export_db`, `GET /settings/backup/export-db`) and `Download Backup` (`backup_create`, `POST /settings/backup/create`) both handed the user the exact same `library.db` file — the only difference was that `backup_export_db` was a bare passthrough with no server-side trace, while `backup_create` also copied the file into `backups/` and called `record_backup()` to keep "Last Backup Date" accurate. Having two buttons for one output was confusing and let someone export via the untracked path and end up with a stale backup-date stat. Removed the untracked duplicate and kept the single action that already did the right thing, renamed for clarity.
- **Database changes:** None.
- **UI changes:** Data & Backup's "Manual Export & Backup" card now shows two actions instead of three: `Export CSV` (unchanged) and `Create Backup` (was `Download Backup` — same underlying behavior, copy → record → download). The button row layout changed from three `col-md-4` columns to two `col-md-6` columns.
- **Future impact:** Recorded ADR-10 in [DECISIONS.md](DECISIONS.md) (single tracked backup/export action, not a tracked+untracked pair) so a second raw-download route isn't reintroduced later. No `TD-N`/`PF-N` opened or resolved — PF-5 (no scheduled backups) still applies, its wording updated to reflect the two remaining actions.
