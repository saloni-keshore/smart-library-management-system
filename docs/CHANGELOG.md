
## 2026-07-23 — Migrated `routes/membership.py` (Membership module) from SQLite to Supabase

- **Feature:** Fifth slice of the incremental app connection-layer cutover to Supabase (ADR-16/17/18/19/20) — `routes/membership.py`'s `index()`, `create()`, and `renew()` now read/write the `memberships` table through Supabase instead of SQLite, with Supabase as the source of truth. Routes, templates, validation, pricing rules (Membership Settings auto-fill), and the renew workflow (expire-old-then-insert-new) are all unchanged. Payments and Cashbook were explicitly excluded from this slice, per this session's scope, even though ADR-19 had named Membership+Payment as a natural pair. Because `memberships.student_id` still enforces a real SQLite foreign key and `routes/payment.py`'s `collect()`, `routes/dashboard.py`, `routes/membership_distribution.py`, and `routes/notification.py` (all explicitly out of scope this session) run raw `JOIN memberships` queries directly against SQLite, `create()`/`renew()` also keep a SQLite mirror row in sync — a per-write mirror, the same shape ADR-18/ADR-19 established for `enquiries`/`students`, not a one-shot bridge like `register()`'s.
- **Files changed:** `routes/membership.py` (every function switched to `database.supabase_client.get_supabase_client()`; `index()` no longer runs the old `EFFECTIVE_STATUS_SQL`-decorated `JOIN students` query against SQLite — it fetches this admin's students from Supabase to build a `student_id → {full_name, mobile}` map, fetches `memberships` scoped to those student ids from Supabase, and merges the two in Python, applying `get_effective_status()` per row; `create()`/`renew()` compute `membership_id` explicitly as SQLite's current `MAX(membership_id) + 1` — not Supabase's auto-assigned identity value, the same fix ADR-18/ADR-19 needed for `enquiry_id`/`student_id` and for the identical reason (see "Why" below) — insert that id into Supabase, then mirror the identical row into SQLite via `database.db.get_connection()`, rolling back the Supabase insert if the SQLite mirror-insert raises `sqlite3.Error`; `renew()` additionally captures which row(s) it's about to flip from `Active` to `Expired` *before* doing so on either database, so a failure partway through the request can restore them instead of leaving a partial expire-without-replace state — a guarantee a single SQLite transaction gave for free before this cutover; both routes now fetch the `student` row from Supabase instead of SQLite; new helper `_sanitize_int()` (same shape as `routes/student.py`'s `_sanitize_date()`) converts `duration_days` to an int or `None` before any Supabase call, since Postgres's `duration_days` column is a strictly-typed `INTEGER` unlike SQLite's), `database/membership_queries.py` (`get_active_membership()` — `create()`'s duplicate-active-membership guard and its only caller anywhere in the codebase, confirmed by grep — now reads Supabase directly instead of the SQLite mirror; every other export, `EFFECTIVE_STATUS_SQL`/`DAYS_LEFT_SQL`/`get_membership_counts`/`get_effective_status`/`get_plan_pricing`/`get_admission_fee`, is untouched and stays SQL/SQLite-backed, since `routes/dashboard.py`/`routes/membership_distribution.py`/`routes/notification.py`/`routes/student.py` all still call them directly against the SQLite mirror), `tests/conftest.py` (new `get_membership_by_id()` helper querying Supabase, mirroring `get_student_by_id()`'s pattern; `get_last_membership_id()` left unchanged — it still works correctly since the SQLite mirror keeps the same `membership_id`), `tests/test_03_student_membership_payment.py` (5 assertions that queried SQLite's `memberships` table directly right after a `create()`/`renew()` call — before any payment collection could touch the SQLite-only `paid_amount`/`pending_amount` fields, see TD-37 — now use `get_membership_by_id()`/a Supabase count query instead; assertions that check state *after* `/payments/collect/<id>` are left on SQLite, since `routes/payment.py` is unmigrated and that route's write only reaches SQLite), `tests/test_09_full_workflow_chain.py` (2 assertions in the renewal chain test switched the same way). Docs: `docs/01_OVERVIEW.md`, `docs/02_ARCHITECTURE.md`, `docs/03_PROJECT_STRUCTURE.md`, `docs/04_DATABASE_SCHEMA.md`, `docs/05_ROUTES_REFERENCE.md`, `docs/09_DEPENDENCY_MAP.md`, `docs/CODE_JOURNEY.md`, `docs/DIAGRAMS.md`, `docs/WHERE_TO_MODIFY.md`, `docs/TROUBLESHOOTING.md`, `docs/FILE_REFERENCE.md` (`database/supabase_client.py`, `database/membership_queries.py`, `routes/membership.py`, `routes/payment.py`, `routes/student.py` notes), `docs/DECISIONS.md` (ADR-20), `docs/11_FUTURE_WORK.md` (TD-37 added, TD-34 updated, "Next logical slices" item 11 added).
- **Why:** requested as the next incremental-migration slice after Auth (ADR-16), Settings' password branch (ADR-17), Enquiries (ADR-18), and Students (ADR-19), scoped explicitly to Membership only — Payments and Cashbook were explicitly excluded from this step. Investigating the dependency graph the same process ADR-16/18/19 used confirmed `memberships.student_id` is a real SQLite FK, and four modules (`routes/payment.py`'s `collect()`, `routes/dashboard.py`, `routes/membership_distribution.py`, `routes/notification.py`) all run raw `JOIN memberships` directly against SQLite — none were touched, per scope, so the SQLite mirror is load-bearing for those four files, not optional scaffolding. `membership_id` needed the same explicit-assignment fix `enquiry_id`/`student_id` needed in ADR-18/19, for the identical reason: Supabase's `memberships` identity sequence was seeded once by the one-time data copy (ADR-15) and had fallen behind SQLite's ever-climbing `AUTOINCREMENT` counter from every session's worth of membership-creation testing since. `get_active_membership()` was moved to Supabase-only reads (rather than continuing to read the SQLite mirror, which stays in sync with every `create()`/`renew()` write in the same request) because it is private to the module being migrated — confirmed by grep, no other file imports it — so the one function `routes/membership.py` uses to guard its own writes has no reason to read a mirror instead of the source of truth it just wrote.
- **Database changes:** None to the schema — `memberships` already existed on Supabase (ADR-14/ADR-15, populated by the one-time `migrate_to_supabase.py` copy). Behavioral change only: `index()`/`create()`/`renew()` now read/write Supabase as the source of truth; SQLite becomes a write-synced mirror for `create()`/`renew()`, kept correct for every column since every consumer of the mirror (`payment.py`, `dashboard.py`, etc.) needs the full row, not a subset. Introduces **TD-37**: `routes/payment.py`'s `collect()` (unmigrated) updates `paid_amount`/`pending_amount` in SQLite only, so Supabase's copy of those two columns goes stale after a payment collection until the same membership is next written by `create()`/`renew()`. `register()`'s SQLite mirror-insert bridge (TD-35) is unaffected — `memberships` was never one of the 7 tables it exists for.
- **UI changes:** None for the happy path — same templates, same forms, same flash messages, same create/renew behavior, same plan-pricing auto-fill. One edge-case behavior change, forced by Postgres's strict typing rather than chosen: submitting a non-numeric/blank `duration_days` now stores `NULL` instead of the literal string SQLite used to accept verbatim (create/renew still succeed either way, no error shown, since `duration_days` isn't read back by any validation or display logic). One new, real staleness case introduced (not a UI bug in this route itself, see TD-37): the Memberships page can show a payment-collection's pre-payment `paid_amount`/`pending_amount` until the same membership's next create/renew.
- **Future impact:** Added **ADR-20** documenting this as the fifth incremental-migration slice and recording the deliberate decision to defer Payment despite ADR-19's pairing suggestion. Added **TD-37** for the new `memberships.paid_amount`/`pending_amount` split-brain this introduced (Open — closing it means migrating `routes/payment.py`, the natural next slice). Updated **TD-34** to note `memberships` as a fourth table where `schema.sql`/`supabase_migration.sql` drift is now a real (not latent) risk. Confirmed `register()`'s SQLite mirror-insert bridge (TD-35) is untouched by this slice — `memberships` was never one of its 7 FK-dependent tables. Confirmed this slice closed one consumer off the `students` SQLite mirror's list (`routes/membership.py`'s own student lookups now go through Supabase), noted in `09_DEPENDENCY_MAP.md`. Final verification: the full 250-test suite (`pytest tests/`, 1 pre-existing skip for TD-30) run against the live Supabase project — all non-skipped tests passed, confirming this slice didn't regress Students, Enquiries, Payments, Cashbook, Dashboard, Notifications, Membership Distribution, Business Intelligence, Settings, or any other already-migrated module.

## 2026-07-23 — Migrated `routes/enquiries.py` (Enquiries module) from SQLite to Supabase

- **Feature:** Third slice of the incremental app connection-layer cutover to Supabase (ADR-16/17/18) — `routes/enquiries.py`'s `index()`, `add()`, `edit()`, `delete()`, and `view()` now read/write the `enquiries` table through Supabase instead of SQLite, with Supabase as the source of truth. Routes, templates, validation, and business logic are unchanged. Because `students.enquiry_id` still enforces a real SQLite foreign key and `routes/student.py`'s `admission()` (explicitly out of scope this session) reads/writes the enquiry row directly against SQLite, `add()`/`edit()`/`delete()` also keep a SQLite mirror row in sync — not a one-shot bridge like `register()`'s, but a per-write mirror, since `admission()` reads live field values (`full_name`/`mobile`/`purpose`/`preferred_shift`), not just the row's existence.
- **Files changed:** `routes/enquiries.py` (every function switched to `database.supabase_client.get_supabase_client()`; `add()` computes `enquiry_id` explicitly as SQLite's current `MAX(enquiry_id) + 1` — not Supabase's auto-assigned identity value, see "Why" below — inserts that id into Supabase, then mirrors the identical row into SQLite via `database.db.get_connection()`, rolling back the Supabase insert if the SQLite mirror-insert raises `sqlite3.Error`; `edit()` updates both, not best-effort, since `admission()` must never read stale pre-edit values; `delete()` deletes from Supabase first — the authoritative, user-visible action — then best-effort deletes the SQLite mirror, swallowing `sqlite3.Error`; `index()`/`view()` additionally query SQLite's `students` table, read-only, to attach `student_id` per enquiry, replacing the old `LEFT JOIN students` SQL with a Python-side merge since `students` isn't on Supabase; new helper `_sanitize_date()` normalizes `followup_date` to `None` on blank/unparsable input before any Supabase call, since Postgres's `DATE` column is strictly typed unlike SQLite's), `tests/conftest.py` (new `get_enquiry_by_id()` helper querying Supabase, mirroring `get_admin_by_username()`'s pattern; `get_last_enquiry_id()` left unchanged — it still works correctly since the SQLite mirror keeps the same `enquiry_id`), `tests/test_02_enquiry.py` (7 assertions that queried SQLite's `enquiries` table directly now use `get_enquiry_by_id()`/a Supabase count query instead; `test_add_enquiry_bad_date_format` tightened from "just doesn't crash" into an explicit assertion that invalid dates are stored as `NULL`, documenting `_sanitize_date()`'s new behavior). Docs: `docs/01_OVERVIEW.md`, `docs/02_ARCHITECTURE.md`, `docs/03_PROJECT_STRUCTURE.md`, `docs/04_DATABASE_SCHEMA.md`, `docs/05_ROUTES_REFERENCE.md`, `docs/09_DEPENDENCY_MAP.md`, `docs/CODE_JOURNEY.md`, `docs/DIAGRAMS.md`, `docs/WHERE_TO_MODIFY.md`, `docs/TROUBLESHOOTING.md`, `docs/FILE_REFERENCE.md` (`routes/enquiries.py`, `routes/student.py`, `database/db.py`, `database/supabase_client.py`, `routes/auth.py`'s `register()` note), `docs/DECISIONS.md` (ADR-18), `docs/11_FUTURE_WORK.md` (TD-34 updated, new TD-36, "Next logical slices" item 9).
- **Why:** requested as the next incremental-migration slice after Auth (ADR-16) and Settings' password branch (ADR-17), scoped explicitly to Enquiries only — "Do not modify Students, Memberships, Payments, Cashbook, or Dashboard in this step." `enquiries` was the natural next candidate: `register()`'s SQLite mirror-insert bridge (TD-35) already listed it as one of 7 FK-dependent tables, and it's a self-contained CRUD module with its own dedicated test file. Investigating the FK dependency (mirroring ADR-16's discovery process for `admins.admin_id`) surfaced that a one-shot `add()`-only bridge, sufficient for `register()`, would not be sufficient here: `routes/student.py`'s `admission()` reads live enquiry field values from SQLite to populate a new student, not just the row's existence, so an enquiry edited after creation (via `edit()`, Supabase-only in a first-draft version of this change) would have fed stale pre-edit data into admission — a real correctness bug, not just a missing-row crash. `edit()` was made to dual-write specifically to close that gap without touching `routes/student.py`. Running the full test suite (not just `test_02_enquiry.py`) surfaced a second, more urgent problem the same way ADR-16's FK issue was found: every single `add()` failed, cascading into 45 failures across `test_02`/`test_03`/`test_04`/`test_05`/`test_07`/`test_09`. Root cause: a first-draft version let Supabase auto-assign `enquiry_id` via its identity column (mirroring `register()`'s pattern), but Supabase's sequence — seeded once by the one-time data copy (ADR-15) — had fallen far behind SQLite's `AUTOINCREMENT` counter (599 vs. 1012, live-verified), which kept climbing through every QA/test session since that copy while `enquiries` was still SQLite-only. Every Supabase-assigned id therefore collided with an id SQLite had already used, the mirror-insert's `PRIMARY KEY` constraint failed every time, and `add()`'s own error path (delete the Supabase row, flash an error) fired on every attempt. No `migrate_reset_identity_sequence` RPC exists in this project to reset the sequence (ADR-15 documented it as optional and it was never created), and this codebase deliberately has no direct-Postgres path for a manual `setval(...)` — so `add()` now computes `enquiry_id` itself from SQLite's current max instead of trusting Supabase's sequence at all, sidestepping the drift permanently (see ADR-18 for the full mechanics).
- **Database changes:** None to the schema — `enquiries` already existed on Supabase (ADR-14/ADR-15, populated by the one-time `migrate_to_supabase.py` copy). Behavioral change only: `index()`/`add()`/`edit()`/`delete()`/`view()` now read/write Supabase as the source of truth; SQLite becomes a write-synced mirror for `add()`/`edit()`/`delete()`, kept correct for `full_name`/`mobile`/`purpose`/`preferred_shift`/`followup_date`/`remarks`/`admin_id`/`enquiry_id` but **not** for `status` — `routes/student.py`'s `admission()` still writes `status='Admitted'` to SQLite only, and nothing propagates that write to Supabase (new debt, TD-36, see below). `register()`'s SQLite mirror-insert bridge (TD-35) is unaffected — `enquiries` remains one of the 7 tables it exists for, since this slice didn't touch `admin_id` row existence, only added a second (per-write, not one-shot) mirror on top.
- **UI changes:** None for the happy path — same templates, same forms, same flash messages, same filtering/list behavior. One edge-case behavior change, forced by Postgres's strict typing rather than chosen: submitting a non-ISO `followup_date` (e.g. `"not-a-date"`) on Add/Edit now stores `NULL` instead of the literal invalid string SQLite used to accept verbatim — the page still saves successfully either way, no error shown to the user.
- **Future impact:** Opened **TD-36** (Medium): `enquiries.status` now has two writers that never sync — `routes/student.py`'s `admission()` writes `'Admitted'` to SQLite only, while the Enquiries pages read `status` from Supabase only — so the Enquiries list/detail views keep showing the pre-admission status and an "Admission" action after a student has actually been admitted, even though the `student_id`/"View Student" link (sourced separately from SQLite) is correct. No test in the existing suite exercises that specific view, so nothing regressed, but it's a real, live gap, not theoretical. - **Feature:** The Payment workflow end-to-end (Student→Membership→Payment→Receipt→Cashbook→Audit Log→Dashboard→BI→Notifications), covering Admission, Renewal, and standalone pending-fee Collect Payment — audited as a Phase 4 follow-up to the Financial System, Dashboard, and Membership Workflow audits, scoped to correctness/consistency/duplicate-logic elimination, no UI redesign, no new features.

- **Why:** live-DB verification (real `library.db`, one admin's actual Receipt Settings: `receipt_prefix="ABC"`, `next_receipt_number=3000`) found the payment-recording path had three separate, drifted implementations and one silently-broken schema link. (1) `routes/payment.py`'s `collect()`, `routes/membership.py`'s `create()`, and `renew()` each inlined their own `INSERT INTO payments` and their own receipt-number formula — `collect()` used `REC-YYYYMMDD-<membership_id>-<int(new_paid)>`, `create()`/`renew()` used `REC-YYYYMMDD-<membership_id>` with no suffix at all — and none of the three read `receipt_prefix`/`next_receipt_number` (TD-22, logged 2026-07-20, confirmed still open after two further audit cycles). (6) `PAYMENT_METHODS` (the canonical list driving Cashbook's payment-method filter dropdown and its manual-entry validation) was missing `"Card"`, even though Payment Collect, Membership Create, and Membership Renew have all offered "Card" as a `payment_mode` option — a Card payment could be recorded but never filtered to in Cashbook. Confirmed via the Flask test client against a scratch copy of the live DB: a POST that the route logic correctly rejected (e.g. `paid_amount=-500`) still rendered a plain 200 form with zero visible indication anything went wrong. This is a pre-existing, wider gap — 11 of the app's ~25 top-level pages do render flashes (Settings sub-pages, Enquiries list, Auth pages), the rest, including several already covered by earlier audits, don't (see TD-31).
- **Database changes:** None — `cashbook.payment_id` and all `library_settings` receipt columns already existed in `schema.sql`/the live DB; this fix only starts populating/reading columns that were already there but unused.
- **UI changes:** Receipt numbers now reflect each admin's configured prefix/starting number (verified: the test admin now gets `ABC-03000`, `ABC-03001`, ... instead of `REC-20260722-...`) instead of ignoring Settings → Receipt Settings entirely. A failed payment/membership save (e.g. an unlikely receipt-number collision) now shows a red flash message with nothing saved, instead of Flask's default error page.
- **Future impact:** Added `database/payment_queries.py` (`generate_receipt_number`, `record_payment`) as the single implementation every payment-creating route now shares (ADR-13) — any future payment-creating flow must call it, not add a fourth inline formula. `insert_income_entry()` gained an optional `payment_id` parameter, now always supplied by `record_payment()` (never by `routes/cashbook.py`'s manual entries, correctly). Verified via a scratch copy of the live DB: receipt allocation, the persisted-counter increment, the `payments`↔`cashbook` `payment_id` link, and end-to-end flash rendering all behave correctly under the Flask test client — including a forced `UNIQUE` collision, which rolled back cleanly with zero orphaned rows in either table, and a repeated double-submit, which (as expected, see TD-30) produced two distinct, individually-valid payments rather than an error. TD-22 flipped to **Resolved (partially — `receipt_prefix`/`next_receipt_number` only)**. Opened TD-29 (`auto_increment_receipt` is read but deliberately not honored — no manual-receipt-number UI exists for "off" to mean anything), TD-30 (no idempotency/double-submit protection on any payment-creating form — each would need a real feature addition, not a quick fix, so both are logged rather than fixed in this audit's correctness/consistency scope), and TD-31 (flash messages invisible on ~14 of the app's ~25 top-level pages, a pre-existing gap this audit's own live-app verification surfaced rather than caught by code reading — partially resolved for the 7 Payment-workflow pages listed above, the rest logged for whichever future audit covers those modules).
