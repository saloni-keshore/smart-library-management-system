# Database Planning

# Smart Library Management System

---

# 1. Introduction

This document defines the database structure of the Smart Library Management System. It identifies all required tables, their relationships, primary keys, foreign keys, and the overall database design principles.

---

# 2. Database Overview

The application uses **MySQL** as the relational database management system.

The database is designed to:

* Store student information
* Manage administrator accounts
* Track memberships
* Record payments
* Store AI predictions
* Generate reports
* Maintain activity logs

---

# 3. Planned Database Tables

The following tables will be created:

1. Admins
2. Students
3. Payments
4. Memberships
5. Predictions
6. Notifications
7. Activity_Logs

---

# 4. Table Descriptions

## 4.1 Admins

Purpose:

Stores administrator login information.

Fields:

* Admin ID (Primary Key)
* Name
* Username
* Email
* Password (Hashed)
* Created Date

---

## 4.2 Students

Purpose:

Stores student details.

Fields:

* Student ID (Primary Key)
* Full Name
* Mobile Number
* Purpose
* Shift
* Admission Date
* Membership Duration
* Membership Expiry
* Status

---

## 4.3 Payments

Purpose:

Stores payment history.

Fields:

* Payment ID (Primary Key)
* Student ID (Foreign Key)
* Amount
* Payment Date
* Payment Method
* Balance
* Payment Status

Relationship:

One Student → Many Payments

---

## 4.4 Memberships

Purpose:

Stores membership details.

Fields:

* Membership ID (Primary Key)
* Student ID (Foreign Key)
* Start Date
* End Date
* Remaining Days
* Membership Status

Relationship:

One Student → One Active Membership

---

## 4.5 Predictions

Purpose:

Stores Machine Learning predictions.

Fields:

* Prediction ID
* Student ID
* Prediction Score
* Risk Category
* Prediction Date

Relationship:

One Student → Many Predictions

---

## 4.6 Notifications

Purpose:

Stores alerts generated by the system.

Fields:

* Notification ID
* Student ID
* Notification Type
* Message
* Created Date
* Status

---

## 4.7 Activity Logs

Purpose:

Maintains system activity history.

Fields:

* Log ID
* Admin ID
* Action
* Module
* Date
* Time

# 6. Primary Keys

Each table uses a unique primary key.

Examples:

* admin_id
* student_id
* payment_id
* membership_id
* prediction_id
* notification_id
* log_id

---

# 7. Foreign Keys

The following foreign key relationships will be used:

* Payments.student_id → Students.student_id
* Memberships.student_id → Students.student_id
* Predictions.student_id → Students.student_id
* Notifications.student_id → Students.student_id
* Activity_Logs.admin_id → Admins.admin_id

---

# 8. Normalization

The database follows Third Normal Form (3NF):

* Eliminate duplicate data
* Store each fact only once
* Use relationships instead of repetition
* Improve consistency

---
