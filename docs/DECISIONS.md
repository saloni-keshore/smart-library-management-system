# Architecture Decisions

Record of deliberate architectural choices and the reasoning behind them, so future changes can be judged against the original intent rather than guessed at. Entries marked **(inferred)** are reconstructed from reading the code/git history, not from a first-hand statement — treat their "why" as a best-effort explanation, not a certainty.

Add a new entry whenever a change reflects a deliberate architectural choice (not just a bug fix) — see the maintenance policy in [README.md](README.md).

---

### ADR-1: Raw SQL via `sqlite3`, no ORM **(inferred)**

**Decision:** All persistence uses the stdlib `sqlite3` module directly (`database/db.py`'s `get_connection()`), with hand-written SQL in route functions and `database/*_queries.py` modules. No SQLAlchemy or other ORM.

**Why (inferred):** SQLite + raw SQL is the simplest thing that works for a single-file, single-server app at this scale, and avoids the setup overhead of an ORM for a project that started as a fast-moving single-developer build (see the "Day 1"/"Day 2" commit messages).

**Consequence:** `models/` and `services/` folders exist but are empty — there's no natural home for ORM model classes or a separate service layer under this architecture. See [08_UTILS_SERVICES_MODELS.md](08_UTILS_SERVICES_MODELS.md).

### ADR-2: `admin_id`-based multi-tenancy retrofitted, not designed in from day one

**Decision:** Tenant isolation is enforced by an `admin_id` column checked manually in each query, added via `database/migrate.py` (commit `90b65e3`, 2026-07-01) — after `enquiries`/`students` already existed without it.

**Why:** the git history shows this landed as a distinct retrofit commit after the initial buildout, not as part of the original schema design.

**Consequence:** isolation is inconsistent across tables — some (`enquiries`, `students`, `library_settings`, `membership_settings`) have a direct `admin_id` column with an FK; some (`cashbook`, `expenses`) have `admin_id` but no FK (added via `ALTER TABLE`, which SQLite can't constrain); `memberships` and `payments` have no `admin_id` at all and rely on a join back to `students`. There is no framework-level enforcement (e.g. row-level security) — a query that forgets the filter would leak cross-tenant data. See [04_DATABASE_SCHEMA.md](04_DATABASE_SCHEMA.md).

**How to apply:** any new admin-scoped table should follow the `enquiries`/`students` pattern (direct `admin_id` column + FK from table creation, not retrofitted later).

### ADR-3: Cashbook as the single source of truth for financial ledger, not `transactions`/`expenses`

**Decision:** All income/expense recording — both manual entries and automatic entries triggered by memberships/payments — flows through the `cashbook` table via `database/cashbook_queries.py`. The `expenses` and `transactions` tables defined in `schema.sql` are not used by any route.

**Why (inferred):** `cashbook` supports both manual and auto-generated entries with a `source` field distinguishing them, plus an audit trail (`audit_log`) — a more general design than the narrower `expenses`/`transactions` tables, which look like earlier or parallel attempts that were superseded once `cashbook` matured (commit `5b482dd`, 2026-07-05).

**Consequence:** `expenses` and `transactions` are dead schema — see [11_FUTURE_WORK.md](11_FUTURE_WORK.md) items 2–3 for the cleanup recommendation (reconcile or drop them so they don't confuse a future contributor into building against the wrong table).

### ADR-4: Auto-generated cashbook entries are read-only in the UI

**Decision:** `routes/cashbook.py`'s `edit_transaction()` explicitly refuses to edit any `cashbook` row whose `source != "Cashbook Manual Entry"`.

**Why (inferred):** rows created by `insert_income_entry()` (from membership/payment flows) are derived from — and must stay consistent with — their originating `payments`/`memberships` row. Allowing direct edits in the Cashbook UI would let the ledger drift out of sync with the record it was generated from.

**How to apply:** if you need to correct an auto-generated entry, fix it at the source (the membership/payment record), not in the Cashbook edit form. Any new auto-generating flow should set a `source` value other than `"Cashbook Manual Entry"` to get this protection automatically.

### ADR-5: Server-rendered matplotlib PNGs for Dashboard/Distribution charts, Chart.js for Cashbook/BI

**Decision:** Two different charting approaches coexist: `utils/charts.py` (matplotlib, server-side, writes a static PNG) for the Dashboard and Membership Distribution pages, versus client-side Chart.js fed by `window.cashbookChartData`/`window.biChartData` for Cashbook and Business Intelligence.

**Why (inferred):** the matplotlib charts (revenue line, membership pie/donut) predate the Chart.js-based pages chronologically (Dashboard/Distribution came earlier in the git history than Cashbook/BI), suggesting the project moved toward client-side interactive charts for newer features while leaving the original server-rendered ones in place rather than back-porting them.

**Consequence:** two genuinely different code paths to maintain, and the matplotlib path currently has the cross-tenant shared-filename issue described in [11_FUTURE_WORK.md](11_FUTURE_WORK.md) item 1 that the Chart.js path doesn't have (since Chart.js data is embedded per-request, not written to a shared file). If unifying charting approaches is ever on the table, Chart.js's per-request data model is the safer default to standardize on.

### ADR-6: No migrations framework — hand-written idempotent scripts instead

**Decision:** Schema changes are one-off Python scripts in `database/migrate_*.py`, each guarding itself with a `PRAGMA table_info` / `CREATE TABLE IF NOT EXISTS` check rather than using Alembic/Flask-Migrate with a version table.

**Why (inferred):** matches the project's overall lightweight, no-framework-beyond-Flask approach; for a single SQLite file with one active environment, a migrations framework's main benefit (coordinating schema state across many environments) doesn't yet apply.

**Consequence:** there's no single command to "bring the DB up to date" — someone has to know to run each new `migrate_*.py` script manually, in the right order, on the right machine. This has already produced one inconsistency (ADR-3's `transactions` table, defined differently in two places depending on which runs first). If the number of environments/deployments grows, revisit this.

### ADR-7: Receipt Settings extends the `library_settings` row instead of a new table, update-only

**Decision:** Receipt numbering/branding/printing preferences (`receipt_prefix`, `next_receipt_number`, `paper_size`, etc.) were added as columns on the existing `library_settings` table rather than a new `receipt_settings` table, and `database/receipt_settings_queries.py`'s `save_receipt_settings()` only supports `UPDATE` — there is no insert/upsert path. `routes/setting.py`'s `receipt_settings()` redirects to `library_profile` if no row exists yet.

**Why:** every receipt setting is inherently tied to one library profile (one row per `admin_id` already existed, one-to-one), and the page reuses that row's `logo_path`/`stamp_path`/`signature_path`/`receipt_footer` directly for its branding preview — splitting it into a second table would mean joining back to `library_settings` on every read for no isolation benefit. Requiring the profile to exist first (rather than allowing an insert with `NULL` `library_name`) avoids violating that column's `NOT NULL` constraint and keeps "receipt settings" from being independently creatable without a library profile behind it.

**Consequence:** an admin must save Library Profile at least once before Receipt Settings becomes usable — this is enforced by the redirect, not by a database constraint. If Receipt Settings is ever split out for reasons unrelated to Library Profile (e.g. a future multi-branch model with one library profile but several receipt configs), this decision would need to be revisited.

### ADR-8: Notification Settings is the single owner of reminder/channel/quiet-hour/dashboard-display preferences

**Decision:** All reminder-day, notification-channel, quiet-hours, and dashboard-display preferences live on the `library_settings` row and are read/written exclusively through `database/notification_settings_queries.py`, surfaced at Settings → Notification Settings. `membership_settings.reminder_days`/`send_reminders` — which previously held the reminder-day/send-reminders concept — are left in the table but `database/membership_settings_queries.py`'s `save_membership_settings()` no longer includes them in its INSERT/UPDATE, and `templates/settings/membership_settings.html` no longer exposes editable inputs for them. Instead, Membership Settings renders a read-only "Reminders & Notifications" summary sourced from `get_notification_settings(admin_id)`, with a link to Notification Settings.

**Why:** the same concept (which reminder days trigger a notification) had drifted into two places — `membership_settings.reminder_days`/`send_reminders` (a single day + on/off flag) and the richer `library_settings.reminder_7_days`/`reminder_3_days`/`reminder_1_day`/etc. added for Notification Settings. Keeping both editable would mean two forms writing to two different representations of the same behavior, with no clear rule for which one wins if they disagree. Notification Settings' representation is strictly more expressive (three independent day-toggles plus expiry-day/after-expiry, versus one day + one flag), so it was picked as the single source of truth; Membership Settings becomes a read-only consumer instead of a second writer.

**Consequence:** the old `membership_settings` columns are now dead weight (see TD-23 in [11_FUTURE_WORK.md](11_FUTURE_WORK.md)) — kept rather than dropped to avoid a destructive schema change for a save-only config table. Anyone extending reminder behavior should treat `library_settings`'s `reminder_*`/`notify_*` columns as the only place to add new reminder-related fields; do not resurrect `membership_settings.reminder_days`/`send_reminders` as a second writable path.

### ADR-9: Data & Backup and Security Settings use dedicated new tables, not `library_settings`

**Decision:** Unlike Receipt Settings and Notification Settings (both of which extend the existing `library_settings` row), Data & Backup uses a new `backup_log` table (one row per admin, tracking `last_backup_at`/`backup_filename`) and Security Settings uses a new `security_settings` table (one row per admin, `session_timeout_minutes`/`remember_me_enabled`/`login_notifications_enabled`), each created by their own `database/migrate_*.py` script and defined directly in `schema.sql` for fresh installs.

**Why:** Receipt Settings and Notification Settings are both legitimately tied to "does this admin have a Library Profile yet" — `receipt_settings()`/`notification_settings()` redirect to `library_profile` if the row doesn't exist, because `library_settings.library_name`/`phone` are `NOT NULL` and there's no sensible lazily-created bare row to insert. Backups and security preferences have no such dependency: an admin should be able to take a backup or set a session timeout before ever filling in a Library Profile. Extending `library_settings` for these would have imported that same "profile must exist first" restriction for no reason, or forced a workaround (allowing `NULL` on columns that are supposed to be required). Separate tables sidestep the coupling entirely — `database/backup_queries.py`'s `record_backup()` and `database/security_settings_queries.py`'s `save_security_settings()` are both plain upserts (`INSERT ... ON CONFLICT(admin_id) DO UPDATE`), no existence guard needed.

**Consequence:** two more single-purpose one-row-per-admin tables to maintain, but no artificial dependency on Library Profile being filled in first. `get_security_settings(admin_id)` returns an in-memory defaults dict (not `None`) when no row exists yet, since there's nothing to redirect to — contrast with `get_notification_settings`/`get_receipt_settings`, which return `None` and rely on the route's redirect-to-`library_profile` guard.

### ADR-10: Data & Backup exposes one backup/export action per distinct output, not one per code path

**Decision:** Data & Backup offers exactly two user-facing actions — `Export CSV` (`backup_export_csv`) and `Create Backup` (`backup_create`) — instead of separately exposing every route that can produce a `.db` file. The `backup_export_db` route (a bare `send_file(DATABASE_PATH)` passthrough with no server-side trace) was removed even though it was technically a distinct code path from `backup_create`, because both produced the same `library.db` content for the user.

**Why:** `backup_export_db` and `backup_create` differed only in bookkeeping — `backup_create` additionally copies the file into `backups/` and calls `record_backup()` to keep `backup_log`'s `last_backup_at` accurate. Exposing both as separate buttons meant a user could download an identical file via the untracked path and be left with a stale "Last Backup Date" stat, with no way to tell from the UI that one button "counts" and the other doesn't.

**How to apply:** if a future Data & Backup addition would produce output that's byte-identical (or near-identical) to an existing action, don't add a new button for it just because the underlying route differs — either fold it into the existing tracked action or make sure it does its own equivalent bookkeeping. A new route/button pair is only warranted when it produces a genuinely different artifact (as `Export CSV` does).

### ADR-11: Fee revenue (Payments) and Cashbook income are distinct metrics, never blended

**Decision:** "Total Revenue" (Dashboard), "Revenue Collected" (Membership Distribution), and the Business Health Score's fee-collection component all derive from `database/cashbook_queries.py`'s new `get_total_fee_revenue(admin_id)` — `SUM(payments.amount_paid)` — instead of `get_total_income(admin_id)` (`SUM(cashbook.amount) WHERE type = 'Income'`), even though both had been used interchangeably in different places before 2026-07-21. `get_pending_fees(admin_id)` was already fee-scoped and is unchanged; `dashboard.py` and `membership_distribution.py` now both call it instead of keeping their own copy of the same `SUM(memberships.pending_amount)` query.

**Why:** an audit of every financial calculation (2026-07-21) found that `get_total_income()` sums *all* Cashbook income, including manual, non-fee categories (`Donation`, `Library Fine`, `Book Sale`, `Membership Card Fee`, `Other Income` — see `database/cashbook_categories.py`'s `MANUAL_INCOME_CATEGORIES`), while "Pending Fees" is specifically `memberships.pending_amount` — money owed against a membership plan. The Business Health Score's collection component computed `1 - pending / (pending + total_income)`, which meant any large non-fee income entry (a real one in this DB: a single mistyped `₹100,000,567` "Library Fine" manual entry for one admin) could drag the ratio to ~100% collected regardless of the library's actual fee-collection performance, because `pending` is fee-only but the denominator wasn't. Dashboard, Membership Distribution, and Cashbook also each computed "how much has this admin been paid" via three different code paths (`SUM(payments.amount_paid)` via SQL, `SUM(memberships.paid_amount)` via SQL, and a Python `sum()` over an already-fetched row list) that happened to agree only because they were mathematically equivalent restatements of the same underlying fact — a maintenance trap, not a guarantee.

**Consequence:** "Total Revenue"/"Revenue Collected" (fee-only) will now read *lower* than Cashbook's "Total Income (All Time)" for any admin who has recorded non-fee manual income — this is correct and intentional, not a regression; the two cards mean different things and are labeled differently for that reason. Anyone adding a new financial KPI that means "money owed/collected against a membership" must call `get_pending_fees`/`get_total_fee_revenue`, not `get_total_income`/a fresh `SUM(payments.amount_paid)` copy — see the updated card for `database/cashbook_queries.py` in [FILE_REFERENCE.md](FILE_REFERENCE.md).

### ADR-12: `database/membership_queries.py` is the single owner of "effective status" and Membership Settings pricing

**Decision:** A new module, `database/membership_queries.py`, now owns: (1) the "effective status" definition — whether a membership row is *really* still active once `end_date` is compared against today, exposed as both a SQL fragment (`EFFECTIVE_STATUS_SQL`) and a Python function (`get_effective_status`); (2) the "days until expiry" expression (`DAYS_LEFT_SQL`); (3) the guard against a student ending up with two simultaneously-`Active` memberships (`get_active_membership`); (4) the active/expired count query shared by Dashboard and Membership Distribution (`get_membership_counts`); and (5) the bridge from `membership_settings` rows to a plan-name-keyed pricing dict for the Create/Renew templates (`get_plan_pricing`/`get_admission_fee`). `routes/membership.py`, `routes/dashboard.py`, `routes/student.py`, `routes/membership_distribution.py`, and `routes/notification.py` all now import from it instead of each carrying their own copy.

**Why:** TD-6 (logged 2026-07-20) predicted that "effective status," independently recomputed in 3 files, was a duplication risk that "could drift." The 2026-07-21 Membership Workflow audit found it already had: `routes/membership.py`'s `index()` (a 4th, previously-unaudited copy) showed a membership as "Active" that every other screen correctly showed as "Expired," because it selected the raw `membership_status` column with no `end_date` comparison at all. Separately, TD-7 (also logged 2026-07-20) had `membership_settings` configured via Settings but never read by Create/Renew — `create.html`/`renew.html` each hardcoded their own `{Monthly:30, Quarterly:90, ...}` JS object, so two templates already disagreed in *style* (one used `Monthly:30`, unquoted; the other `"Monthly":30`) even though their values happened to still match. Centralizing both concerns in one queries module (matching the established `cashbook_queries.py`/`bi_queries.py` pattern) closes both gaps the same way: one definition, multiple call sites, instead of N definitions that only agree by coincidence.

**Consequence:** `EFFECTIVE_STATUS_SQL` is a raw SQL fragment meant for f-string interpolation into a query that already has a `memberships m` alias in scope — it is never a bind parameter. When it's selected under an alias that collides with an existing `m.*` column (as in `routes/membership.py`'s `index()` and `routes/student.py`'s `index()`), it must be listed *before* `m.*` in the `SELECT` list, because `sqlite3.Row` resolves a duplicate column name to whichever occurrence came first (verified empirically, not documented behavior) — getting the order backwards would silently reintroduce the exact bug this ADR fixes. `create()` now redirects to `renew()` instead of inserting a second membership when `get_active_membership()` finds one already live — any future entry point that creates a membership must call this guard too. Plan pricing/duration are auto-filled from settings but remain admin-editable (not server-enforced) — a discount or a one-off custom fee is still possible, this only fixes the *default* no longer being a hardcoded, settings-oblivious number.

### ADR-13: `database/payment_queries.py` is the single owner of receipt numbering and payment recording

**Decision:** A new module, `database/payment_queries.py`, now owns every write to the `payments` table. `generate_receipt_number(conn, admin_id)` allocates the next receipt number from `library_settings.receipt_prefix`/`next_receipt_number`, advancing that counter in the same transaction as the payment. `record_payment(conn, admin_id, ...)` inserts the `payments` row and its matching automatic Cashbook Income entry — now passing `payment_id` through to `insert_income_entry()` so `cashbook.payment_id` (declared as a FK in `schema.sql` since before this ADR, but never populated) actually links back to the payment that produced it. `routes/payment.py`'s `collect()` and `routes/membership.py`'s `create()`/`renew()` all call `record_payment()` instead of each inlining their own `INSERT INTO payments` + receipt-number formula.

**Why:** the 2026-07-22 Payment Workflow Audit found the three payment-creating routes had already drifted into two incompatible receipt-number formats — `collect()` used `REC-YYYYMMDD-<membership_id>-<new_paid>`, while `create()`/`renew()` used `REC-YYYYMMDD-<membership_id>` with no amount suffix — neither of which read the `receipt_prefix`/`next_receipt_number`/`auto_increment_receipt` an admin configures on Settings → Receipt Settings (TD-22, logged 2026-07-20, still open a full audit cycle later). `collect()`'s format was also a real collision risk: truncating a running `float` total with `int()` for the suffix meant two partial payments made the same day on the same membership could land on the same integer (e.g. paid totals of `500.4` and `500.8` both truncate to `500`), producing a `UNIQUE` constraint violation on `payments.receipt_number` that no route caught — an unhandled `sqlite3.IntegrityError` on a payment. Separately, `database/migrate_backfill_cashbook_payments.py` (a real, already-shipped script) could only match a `payments` row to its Cashbook entry by `(admin_id, person, amount, entry_date)` — proof by construction that reconciling "did this payment produce exactly one ledger entry" was never actually verifiable via the FK the schema declares, only by a fuzzy heuristic that two payments with the same person/amount/date would defeat.

**Consequence:** Receipt numbers are now strictly sequential per admin and reflect their configured prefix/starting number — an admin who set `receipt_prefix="ABC"`, `next_receipt_number=3000` now actually gets `ABC-03000`, `ABC-03001`, ... instead of a `REC-...` string their settings never touched. `generate_receipt_number()` always advances the counter regardless of `auto_increment_receipt`, because no manual-receipt-number entry field exists anywhere in the UI — honoring "off" as "don't advance" would make every payment after the first fail on the `UNIQUE` constraint for that admin. That toggle remains inert until a manual-entry UI is built (still no worse than before this ADR, where it was inert for a different reason — nothing read it at all). `routes/payment.py`/`routes/membership.py` now wrap their payment-recording sequence in `try/except sqlite3.Error: conn.rollback()`, so the rare remaining collision case (or any other DB error mid-write) surfaces as a flashed error with nothing saved, not a 500 page with an already-updated `memberships` row and no matching `payments` row. Any future third payment-creating flow must call `record_payment()`, not reintroduce a fourth inline formula.

### ADR-14: Supabase (PostgreSQL) migration script hand-translated as a literal mirror of `schema.sql`, not a redesign

**Decision:** `database/supabase_migration.sql` is a line-for-line PostgreSQL translation of `database/schema.sql`, intended to be run once in the Supabase SQL Editor to stand up an equivalent schema there. Translation choices were kept mechanical: `INTEGER PRIMARY KEY AUTOINCREMENT` → `INTEGER PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY` (`BY DEFAULT`, not `ALWAYS`, so a future data-migration script can insert explicit legacy IDs); `REAL` → `DOUBLE PRECISION` (SQLite's `REAL` is an 8-byte IEEE float, PostgreSQL's own `REAL` is only 4-byte, so `DOUBLE PRECISION` is the precision-preserving translation, not `REAL`); `PRAGMA` statements dropped (no PostgreSQL equivalent); the `cashbook` table's six `ALTER TABLE ... ADD COLUMN` statements folded into its `CREATE TABLE`. No table was added, removed, renamed, or given new constraints/columns beyond what `schema.sql` already declares.

**Why:** `requirements.txt` already lists the `supabase` client library and a connectivity smoke test (`test_supabase.py`) exists, indicating a Supabase migration is underway. The ask was explicitly to translate, not redesign — so every type mapping above was chosen to be the *faithful* equivalent of the existing SQLite type/constraint, even where a "nicer" PostgreSQL-native choice exists (e.g. `NUMERIC` for money instead of `DOUBLE PRECISION`, or `BOOLEAN` instead of `INTEGER` for the many 0/1 flag columns) — those would be schema redesign decisions, deliberately left for a future ADR if the project decides to make them.

**Consequence:** The app itself (`database/db.py`, every `routes/*.py`, every `database/*_queries.py`) still talks to SQLite via `sqlite3` — this script only creates a parallel, currently-unused PostgreSQL schema. Until a connection-layer cutover happens, `schema.sql` and `supabase_migration.sql` are two hand-maintained copies of the same design that **will drift** if one is edited without the other — the same category of risk ADR-3/TD-2 already describes for `expenses`/`transactions`. See TD-34. The many `INTEGER` 0/1 "boolean" columns (`demo_done`, `auto_expiry`, `remember_me_enabled`, etc.) and financial `REAL`/`DOUBLE PRECISION` columns are prime candidates to reconsider (`BOOLEAN`, `NUMERIC`) at actual cutover time, once the app's queries are being rewritten anyway and a real redesign is in scope — not before.

### ADR-15: `migrate_to_supabase.py` uses the Supabase Python client only, with a print-SQL fallback for identity-sequence resets

**Decision:** `database/migrate_to_supabase.py` copies `library.db`'s rows into the Supabase tables created by ADR-14's `supabase_migration.sql` using only the `supabase`/`postgrest` Python client (REST over HTTP), matching the pattern `test_supabase.py` already established — no `psycopg2`/direct Postgres connection string was added. Since PostgREST has no endpoint for arbitrary SQL, resetting each table's identity sequence after preserving legacy primary keys (`setval(...)`) can't be done through the client alone. The script instead calls an *optional* RPC, `migrate_reset_identity_sequence(p_table, p_id_column, p_value)`, and if that function doesn't exist in the target project (it does not, by default — creating it would itself be a schema change, out of this script's scope), it prints the exact `setval(pg_get_serial_sequence(...), ...)` statements for the user to paste into the Supabase SQL Editor once, at the end of the migration report.

**Why:** the task that produced this script explicitly scoped it to "only create `database/migrate_to_supabase.py` and any minimal documentation," ruling out adding a new dependency (`psycopg2`) or touching the schema (creating the helper RPC server-side). The Supabase client's REST-only nature is a real, permanent constraint, not an oversight — better to surface it honestly (print exact, copy-pasteable SQL) than to silently leave sequences unreset or fabricate a connection method the project doesn't otherwise use.

**Consequence:** Data import and row-count verification never depend on the sequence-reset step succeeding — a table can be fully migrated and verified while still needing a manual one-line SQL statement afterward. If a project later adds a generic SQL-exec RPC (or `migrate_reset_identity_sequence` specifically) to Supabase for other tooling, this script will pick it up automatically and stop printing manual fallback SQL for that table, with no code change required.

### ADR-16: App connection-layer cutover to Supabase happens incrementally, module by module — `routes/auth.py` first

**Decision:** Rather than cutting every route over to Supabase in one change, the app's connection layer is being migrated one module at a time, starting with `routes/auth.py` (login/register/forgot-password). A new reusable client, `database/supabase_client.py`'s `get_supabase_client()` (cached with `functools.lru_cache`, mirroring `database/db.py`'s `get_connection()` role but for Supabase), replaces `database.db.get_connection()` in that one file only. Every other route/`database/*_queries.py` module is untouched and continues reading/writing SQLite exactly as before. Routes, templates, business logic, and validation in `routes/auth.py` are unchanged — only the persistence calls were swapped, `.eq()`-filtered against the `admins` table Supabase already has (via ADR-14/ADR-15's schema translation and one-time data copy).

**Why:** a single big-bang cutover of every module at once is high-risk (13 blueprints, ~30 templates, no query-module abstraction for several of them) and hard to verify incrementally. `routes/auth.py` is the smallest, most self-contained candidate — it touches exactly one table (`admins`), has its own dedicated test file (`tests/test_01_auth.py`), and its own `tests/conftest.py`'s `new_admin` fixture already isolates every other test suite from needing to know how admin accounts are created. Cutting it over first, verifying the full test suite still passes, and only then moving to the next module keeps each step independently reviewable and revertible.

**Consequence — a real, temporary split-brain on `admins.password` (see TD-35):** `routes/auth.py`'s `forgot_password()` updates `admins.password` in Supabase only. `routes/setting.py`'s `security_settings()` (Settings → Security Settings → Change Password) still updates it in SQLite only — that route wasn't touched, per this same incremental-scope decision. Until Settings is migrated too, a password changed via Settings has no effect on login (login checks Supabase's copy, which the SQLite-only write never reached), and a password reset via `forgot_password()` is likewise invisible to `security_settings()`'s "current password" check (still SQLite). This is a known, accepted cost of doing the migration incrementally rather than all at once — the alternative (migrating `routes/setting.py`'s password-change branch in the same change) was ruled out to keep this step scoped to the auth module only, matching the stated incremental-migration plan.

**Consequence — `register()` needed a SQLite mirror-write, not a pure Supabase swap, to keep the rest of the app from breaking for new admins:** unlike `login()`/`forgot_password()`, `register()` creates a brand-new `admin_id`. Seven other still-SQLite tables enforce a real SQLite foreign key back to `admins.admin_id` (confirmed directly in `database/schema.sql`, not inferred): `enquiries` (`routes/enquiries.py`), `students` (`routes/student.py`), `audit_log` (written transitively from three modules — `routes/cashbook.py`, `routes/membership.py`, and `routes/payment.py` all eventually call `database/audit_queries.py`'s `log_entry()` via `database/cashbook_queries.py`), and `library_settings`/`membership_settings`/`backup_log`/`security_settings` (all four owned by `routes/setting.py`). `cashbook`/`expenses` also carry an `admin_id` column but were added via `ALTER TABLE ADD COLUMN`, which SQLite can't attach a `FOREIGN KEY` to, so those two don't enforce anything. `database/db.py`'s `get_connection()` sets `PRAGMA foreign_keys = ON` on every connection, so all 7 are actively enforced, not latent. Running the full test suite against the initial pure-Supabase version of `register()` surfaced this immediately: 74 failures, all `sqlite3.IntegrityError: FOREIGN KEY constraint failed` or its downstream `database is locked` symptom, every one traced back to a newly-registered admin (via `tests/conftest.py`'s `new_admin` fixture) that existed in Supabase but not in SQLite. In production this specific path is rare — `register()` already flashes "available only during initial setup" once any admin exists — but it would still hit a fresh deployment's very first admin at the worst possible moment: signup succeeds, then every other feature they try immediately breaks. Given this session's explicit "ensure the app remains runnable after each step" requirement, `register()` was changed to insert into Supabase first (the new source of truth for login), then mirror the same row — same `admin_id`, `full_name`, `username`, `mobile`, `email`, hashed `password`, `role` — into SQLite's `admins` table, rolling back the Supabase insert if the SQLite mirror fails (`try/except sqlite3.Error`, `supabase.table("admins").delete().eq("admin_id", ...)`). `login()` and `forgot_password()` remain pure Supabase reads/writes — this mirror is specific to `register()`'s FK-creation problem, not a general both-databases policy.

**Also new in this change:** every Supabase call in `routes/auth.py` that carries user-controlled input is wrapped in `try/except postgrest.exceptions.APIError`. This isn't new business logic — it's error handling for a failure mode SQLite never had: PostgREST encodes `.eq()`/`.select()` filter values into the request's GET query string, and Supabase's Cloudflare-fronted edge network was observed, live, to return an HTTP 403 WAF block for adversarial-looking query strings (confirmed via `tests/test_01_auth.py`'s existing SQL-injection-payload tests, e.g. a username containing `DROP TABLE`) before the request reaches Postgres at all. Without the `try/except`, that 403 surfaces as an unhandled `APIError` → Flask's generic 500 handler — worse than SQLite's graceful "Invalid username" for the same input. The `except` blocks degrade to the same outcome the equivalent "no match" branch already produced (or a generic "Something went wrong" flash for `register()`'s existence-check/insert calls), so normal (non-adversarial) behavior is completely unchanged. Actual SQL-injection safety is unaffected either way — `.eq()` values are never string-interpolated into a query, so injection was never possible; this is purely about not crashing on input that happens to look like an injection attempt.

**How to apply:** the next module to migrate should follow the same shape — add/extend `database/supabase_client.py` usage in that module only, leave every other module on SQLite, run the full test suite (not just that module's own tests — the FK-mirror issue above was only caught by running everything), and update this table's status in [04_DATABASE_SCHEMA.md](04_DATABASE_SCHEMA.md) and [11_FUTURE_WORK.md](11_FUTURE_WORK.md)'s TD-34/TD-35 rows. If `routes/setting.py`'s `security_settings()` is the next slice (closing TD-35's password-sync half), it should reuse `database/supabase_client.py`'s `get_supabase_client()`, not roll its own client. Note that migrating all of `routes/setting.py` only closes 4 of the 7 FK dependencies (`library_settings`/`membership_settings`/`backup_log`/`security_settings`) — `enquiries`, `students`, and `audit_log` belong to three separate modules (Enquiries, Students, and Cashbook/Membership/Payment respectively) and must each migrate too before `register()`'s SQLite mirror-insert can be deleted. Re-check after each of those migrations whether any FK dependency remains before removing the mirror — removing it early would reintroduce the exact `sqlite3.IntegrityError: FOREIGN KEY constraint failed` this ADR fixed.

### ADR-17: Settings module's second Supabase slice is scoped to the password-change branch only — the rest of `routes/setting.py` stays on SQLite

**Decision:** Following ADR-16's incremental plan, `routes/setting.py`'s `security_settings()` password-change branch (`form_type == "password"`) now reads/writes `admins.password` via `database/supabase_client.py`'s `get_supabase_client()`, reusing the exact client `routes/auth.py` already uses — no new client code. Nothing else in `routes/setting.py` was touched: `library_profile()`, `membership_settings()`, `receipt_settings()`, `notification_settings()`, `staff_access()`, `data_backup()`/`backup_export_csv()`/`backup_create()`, and `security_settings()`'s own session-preferences half (`session_timeout_minutes`/`remember_me_enabled`/`login_notifications_enabled`, saved via `database/security_settings_queries.py`) all continue reading/writing SQLite exactly as before. Routes, templates, business logic, and validation are unchanged — the current-password check, new/confirm-password match check, and `routes.auth.validate_password` strength check all run identically; only the two `admins` table calls (`SELECT`/`UPDATE`) were swapped from `sqlite3` to Supabase `.eq()` filters, wrapped in `try/except postgrest.exceptions.APIError` matching `routes/auth.py`'s existing pattern for the same table.

**Why:** this is the natural second slice ADR-16's "How to apply" section named directly — same table (`admins`), same client, same shape as the auth cutover, and it closes TD-35's `admins.password` split-brain (a password changed via Settings previously had no effect on login, since login already reads Supabase). Scoping it to only the password branch (not all of `routes/setting.py`) keeps the change small and independently verifiable, consistent with ADR-16's "each step independently reviewable and revertible" rationale. The other four tables `routes/setting.py` owns (`library_settings`, `membership_settings`, `backup_log`, `security_settings` itself) have no `admins`-style split-brain risk — nothing else reads or writes them from a second location the way `login()`/`forgot_password()`/`security_settings()` all touched `admins.password` — so migrating them now would be scope creep against this session's explicit "Settings module... only" instruction, not a fix for anything broken.

**Consequence — `register()`'s SQLite mirror-insert bridge (TD-35) is unaffected and still fully required:** this slice changed `admins.password`'s writer, not `admins.admin_id`'s existence. The 7 tables with a real SQLite FK to `admins.admin_id` (`enquiries`, `students`, `audit_log`, and `routes/setting.py`'s own `library_settings`/`membership_settings`/`backup_log`/`security_settings`) are untouched by this change — `register()` must keep mirroring new admin rows into SQLite until all 7 (not just the password column) are migrated. Migrating Settings' password branch closes 0 of those 7 FK dependencies; it only closes the separate `admins.password` value-sync problem TD-35 also described.

**How to apply:** the next slice should pick one of the remaining SQLite-only tables/modules (e.g. `routes/setting.py`'s `library_settings`/`membership_settings`/`backup_log`, or `routes/enquiries.py`'s `enquiries`, or `routes/student.py`'s `students`) and repeat this same pattern — reuse `database/supabase_client.py`, touch only that table's read/write calls, leave everything else (routes, templates, validation, other tables) untouched, and run the full suite afterward. Only once all 7 FK-dependent tables are migrated can `register()`'s SQLite mirror-insert be safely deleted — check the remaining count in TD-35 before removing it.

### ADR-18: Enquiries is the third Supabase slice — and the first one that needed its own two-way SQLite mirror, not just a one-time bridge insert

**Decision:** `routes/enquiries.py` (`index()`, `add()`, `edit()`, `delete()`, `view()`) now reads/writes the `enquiries` table via `database/supabase_client.py`'s `get_supabase_client()`, the same client `routes/auth.py` (ADR-16) and `routes/setting.py` (ADR-17) use. Supabase is the source of truth for every read (`index()`'s list, `edit()`'s/`view()`'s single-row fetch). Unlike `register()`'s one-shot mirror-insert, this slice needed a mirror kept in sync across **three** write paths, not one: `add()` mirrors the new row (same `enquiry_id`) into SQLite; `edit()` updates Supabase then applies the identical update to the SQLite mirror; `delete()` deletes from Supabase (authoritative) then best-effort deletes the SQLite mirror row, swallowing `sqlite3.Error` rather than rolling back the Supabase delete. `index()` still needs a `students` lookup to attach `student_id` per enquiry the way the old `LEFT JOIN students s ON s.enquiry_id = e.enquiry_id` did — `students` stays in SQLite (out of scope this session), so `index()` fetches Supabase's enquiries, separately queries SQLite for this admin's `enquiry_id → student_id` pairs, and merges them in Python instead of a database-level join. Routes, templates, validation, and business logic are otherwise unchanged; `followup_date` gained a small new `_sanitize_date()` normalization step (see the "Why" below) but every other field is untouched.

**`add()`'s `enquiry_id` is computed explicitly, not taken from Supabase's auto-assigned identity value:** a first-draft version of this change let Supabase's `GENERATED BY DEFAULT AS IDENTITY` column assign `enquiry_id` (mirroring `register()`'s pattern for `admin_id`), then mirrored that value into SQLite. Running the full test suite immediately surfaced every single `add()` as a failure: Supabase's `enquiries` identity sequence was seeded once by the one-time data copy (ADR-15) and had advanced only via this session's own testing, while SQLite's `AUTOINCREMENT` counter had kept climbing across every prior QA/test session since that copy (`enquiries` was still SQLite-only until today) — live-verified at `enquiry_id` 599 on Supabase's side vs. 1012 already used in SQLite. Supabase therefore kept re-issuing ids SQLite had already claimed, the SQLite mirror-insert's `PRIMARY KEY` collided every time, and `add()`'s own rollback (delete the Supabase row, flash an error) fired on every attempt — cascading into 45 failures across `test_02`/`test_03`/`test_04`/`test_05`/`test_07`/`test_09` (anything that calls the `make_enquiry` test helper). No `migrate_reset_identity_sequence` RPC exists in this project (ADR-15 already documented this as optional, and it was never created), and this codebase deliberately has no direct-Postgres/`psycopg2` path to run a manual `setval(...)` — so `add()` was changed to compute `enquiry_id` itself, as `SELECT MAX(enquiry_id) FROM enquiries` (SQLite) `+ 1`, and pass that explicit value to the Supabase insert (legal specifically because the column is `GENERATED BY DEFAULT`, not `ALWAYS` — see `supabase_migration.sql`'s translation notes). SQLite's max is always the safe upper bound going forward too, since every future `add()` writes the identical id to both databases in the same request, keeping them in lockstep from here on — the one-time gap only needed bridging once, at the first synchronized insert.

**Why:** `students.enquiry_id` has a real SQLite foreign key to `enquiries.enquiry_id` (`database/db.py` sets `PRAGMA foreign_keys = ON` on every connection), exactly like `admins.admin_id`'s FK problem in ADR-16 — but here the dependency is deeper than `register()`'s one-shot insert. `routes/student.py`'s `admission()` (explicitly out of this session's scope — "Do not modify Students... in this step") does three things directly against SQLite: reads the full enquiry row (`full_name`/`mobile`/`purpose`/`preferred_shift`) to populate the new student, inserts the student row with a real FK to `enquiry_id`, and afterward runs `UPDATE enquiries SET status='Admitted' WHERE enquiry_id=?`. Since that route can't be touched, `add()` alone (a one-shot mirror like `register()`'s) wasn't enough — if an admin edited an enquiry's mobile number after creation and `edit()` only wrote to Supabase, `admission()` would silently admit the student with the stale pre-edit mobile number read from SQLite, a real correctness bug, not just a missing-row crash. Making `edit()` (and, for symmetry/cleanliness, `delete()`) dual-write closes that specific gap without touching `routes/student.py`. Postgres's `followup_date DATE` column is strictly typed (unlike SQLite's dynamic typing, which silently accepted `""`/`"not-a-date"` and stored it verbatim) — `_sanitize_date()` converts blank/unparsable input to `NULL` before the Supabase call instead of letting `postgrest.exceptions.APIError` (invalid date syntax) surface as an unhandled 500, the same category of new failure mode ADR-16 already documented for adversarial-looking query strings.

**Consequence — a new, real split-brain this slice could not close (TD-36):** `admission()`'s `UPDATE enquiries SET status='Admitted' ...` is SQLite-only and untouched by this change. After a student is admitted, the SQLite mirror's `status` becomes `"Admitted"`, but Supabase's copy — the one `index()`/`edit()`/`view()` now actually read — stays whatever it was before (`"Interested"` by default). The Enquiries list/detail pages will therefore keep showing the pre-admission status and the "Admission" action instead of flipping to the "Admitted" badge and "View Student" link, even though `student_id` (read from SQLite, unaffected) correctly shows the enquiry has in fact been admitted. No test in this session's suite exercises "view Enquiries after admission," so nothing regressed there, but this is a real, user-visible gap, not a theoretical one — see TD-36. This is structurally the same category of consequence as TD-35 (ADR-16/17), just surfacing on a `status` column instead of `password`.

**Consequence — `register()`'s SQLite mirror-insert bridge is now partially, not fully, redundant for `enquiries` specifically, but still required:** `enquiries` is one of the 7 tables `register()`'s bridge exists for. This slice doesn't let that bridge be deleted — a newly registered admin's SQLite row is still needed for `add()`'s own mirror-insert to satisfy `enquiries.admin_id`'s SQLite FK, and `students`/`audit_log`/the rest of `routes/setting.py`'s tables are still unmigrated. Track the remaining count in TD-35.

**How to apply:** the next slice should be `routes/student.py`'s `students` table — migrating it would let `admission()`'s three SQLite operations move to Supabase too, closing TD-36's status split-brain at the source (rather than by, say, having `routes/enquiries.py` poll for status changes, which would be a workaround, not a fix) and removing one more of `register()`'s 7 FK dependencies. Until then, anyone touching `admission()` for an unrelated reason should know it's reading a SQLite mirror that `routes/enquiries.py` keeps in sync only for `full_name`/`mobile`/`purpose`/`preferred_shift`/`followup_date`/`remarks` — not for `status`, which flows the other direction (SQLite → nowhere, since Supabase is never told).

### ADR-19: Students is the fourth Supabase slice — closes TD-36 at the source, but the widest fan-out of unmigrated dependents seen so far

**Decision:** `routes/student.py` (`index()`, `admission()`, `view()`, `edit()`) now reads/writes the `students` table via `database/supabase_client.py`'s `get_supabase_client()`, the same client `routes/auth.py` (ADR-16), `routes/setting.py` (ADR-17), and `routes/enquiries.py` (ADR-18) already use. Supabase is the source of truth for every read. Like `enquiries` (ADR-18) and unlike `register()`'s one-shot mirror-insert, this slice needed an ongoing, per-write SQLite mirror: `admission()` mirrors the new row (same explicit `student_id`, computed as `SELECT MAX(student_id) FROM students` (SQLite) `+ 1` for the identical reason ADR-18 needed this for `enquiry_id` — Supabase's identity sequence, seeded once by the one-time data copy, ADR-15, trails SQLite's ever-climbing counter); `edit()` updates both, not best-effort. `admission()` also now reads/writes the `enquiries` row it needs directly against Supabase (via the same client) instead of the SQLite mirror `routes/enquiries.py` maintained for it — its guard-check read (`SELECT * FROM enquiries WHERE enquiry_id=? AND admin_id=?`) and its `status='Admitted'` write both now target Supabase, the copy `routes/enquiries.py` itself reads for the Enquiries pages. `index()` no longer runs the old correlated-subquery `LEFT JOIN memberships` against `students` in one query — it fetches Supabase's students list, separately queries SQLite for this admin's latest-membership-per-student (`EFFECTIVE_STATUS_SQL`, unchanged), and merges the two in Python, the same shape ADR-18 used for `enquiries.index()`'s `student_id` merge. `edit()`'s duplicate-mobile handling changed from a post-write `sqlite3.IntegrityError` catch to a pre-write existence check against Supabase (mirroring `routes/auth.py`'s `register()` username/mobile check), since a Supabase unique-violation can't be told apart from any other `postgrest.exceptions.APIError` without parsing its error code — check-first is the more direct fit for the new backend, and preserves the exact same user-facing message. `join_date` gained a `_sanitize_date()` normalization step, identical in shape to `routes/enquiries.py`'s `followup_date` handling, for the same reason (Postgres's `DATE` column is strictly typed, unlike SQLite's).

**Why:** requested as the next incremental-migration slice, explicitly naming Students and asking for a determination of whether Enquiries' SQLite mirror was still needed afterward. This matches ADR-18's own "How to apply" recommendation: `students` closes TD-36 at the source (rather than a workaround on the Enquiries side), and is one of `register()`'s 7 FK-dependent tables. Investigating the dependency graph the same way ADR-16/ADR-18 did surfaced a materially wider fan-out than `enquiries` had: `memberships.student_id` is a real SQLite FK (`database/db.py` sets `PRAGMA foreign_keys = ON` on every connection), and **eight** other modules run raw `JOIN students`/`SELECT * FROM students` directly against SQLite — `routes/membership.py`, `routes/payment.py`, `routes/dashboard.py`, `routes/membership_distribution.py`, `routes/notification.py`, `routes/setting.py`'s `backup_export_csv()`/`backup_create()`, `database/bi_queries.py`, `database/cashbook_queries.py`, and `database/membership_queries.py` — none of which this session's scope permitted touching. This is why the SQLite mirror could not be a one-shot bridge (as `register()`'s is) or dropped once Supabase became the source of truth for `routes/student.py` itself: it is genuinely load-bearing for eight other files' correctness, not transitional scaffolding for `students` alone.

**Post-migration determination on the Enquiries SQLite mirror (explicitly requested):** the mirror `routes/enquiries.py` maintains (ADR-18) is **still required**, for two reasons unrelated to `students` migrating: (1) `routes/dashboard.py`'s `index()` runs a plain `SELECT COUNT(*) FROM enquiries` against SQLite for its "Total Enquiries" KPI — untouched, out of this session's scope, and it would silently under/over-count if the mirror stopped being written; (2) `routes/enquiries.py`'s own `index()`/`view()` still query SQLite's `students` table (not Supabase) to build the `enquiry_id → student_id` map — also untouched. What this slice *did* change is which write reaches Supabase: `admission()` now updates `enquiries.status` in Supabase directly, instead of the SQLite mirror only, closing TD-36's split-brain — but the mirror's row existence and its other columns (`full_name`/`mobile`/`purpose`/`preferred_shift`/`followup_date`/`remarks`, kept in sync by `routes/enquiries.py`'s own `add()`/`edit()`) remain exactly as necessary as ADR-18 left them, for exactly the reasons ADR-18 gave — this slice neither removed nor added a dependency on that data.

**Consequence — TD-36 is now `Resolved`:** `enquiries.status` has a single writer again (`admission()`, writing Supabase directly), the same shape as TD-35's `admins.password` resolution (ADR-17). The Enquiries list/detail pages now correctly reflect a just-completed admission's status and offer "View Student" instead of a stale "Admission" action.

**Consequence — `register()`'s SQLite mirror-insert bridge (TD-35) is unaffected and still fully required, and the `students`/`enquiries` SQLite mirrors themselves are not yet removable:** `students` is one of the 7 tables `register()`'s bridge exists for; this slice doesn't let that bridge be deleted, for the identical reason ADR-18's `enquiries` slice didn't — `admission()`'s own SQLite mirror-insert still needs `students.admin_id`'s SQLite FK to resolve, so a new admin's SQLite row is still required. `audit_log` and the rest of `routes/setting.py`'s tables (`library_settings`/`membership_settings`/`backup_log`/`security_settings`) remain unmigrated and still require it too. Separately — and distinctly from the `admins.admin_id` FK question — the `students`/`enquiries` SQLite *mirrors* (as opposed to the `admins` mirror-insert bridge) can only be deleted once every one of the modules named in the "Why" section above (for `students`) and Dashboard's enquiry count / Enquiries' own student lookup (for `enquiries`) are migrated too; that is a materially larger remaining slice than the `admins.admin_id` FK question alone.

**How to apply:** the next slice should pick one of the remaining SQLite-only tables (`audit_log`, `library_settings`, `membership_settings`, `backup_log`, `security_settings` — all four of the latter owned by `routes/setting.py`) to close more of `register()`'s 7 FK dependencies, or, if reducing the `students`/`enquiries` SQLite mirrors' footprint is the goal instead, migrate one of the eight modules named above (Membership and Payment are the natural pair, since they already share `database/payment_queries.py`/`database/membership_queries.py` and would let `memberships`/`payments` move to Supabase together) — check which of `students`'/`enquiries`' remaining SQLite consumers that would close before assuming either mirror can be trimmed.

### ADR-20: Membership is the fifth Supabase slice — Payment deliberately deferred despite ADR-19's pairing suggestion

**Decision:** `routes/membership.py` (`index()`, `create()`, `renew()`) now reads/writes the `memberships` table via `database/supabase_client.py`'s `get_supabase_client()`, the same client `routes/auth.py` (ADR-16), `routes/setting.py` (ADR-17), `routes/enquiries.py` (ADR-18), and `routes/student.py` (ADR-19) already use. Supabase is the source of truth for every read. Like `enquiries`/`students` and unlike `register()`'s one-shot mirror-insert, this slice needed an ongoing, per-write SQLite mirror: `create()`/`renew()` mirror the new/updated row (same explicit `membership_id`, computed as `SELECT MAX(membership_id) FROM memberships` (SQLite) `+ 1`, the same fix ADR-18/ADR-19 needed for `enquiry_id`/`student_id` since Supabase's identity sequence — seeded once by the one-time data copy, ADR-15 — trails SQLite's ever-climbing counter). `index()` no longer runs the old `EFFECTIVE_STATUS_SQL`-decorated `JOIN students` query directly against SQLite — it fetches this admin's students from Supabase (`routes/student.py`, ADR-19) to build a `student_id → {full_name, mobile}` map, fetches `memberships` scoped to those student ids from Supabase, and merges the two in Python, computing effective status per row with `database/membership_queries.py`'s existing `get_effective_status()` — the same cross-table-merge shape ADR-18/ADR-19 used for `enquiries.py`'s/`student.py`'s own merges in the other direction. `database/membership_queries.py`'s `get_active_membership()` — `create()`'s duplicate-active-membership guard, and (confirmed by grep) its only caller anywhere in the codebase — now reads Supabase directly too, instead of the SQLite mirror the rest of that module still serves; every other export in that module (`EFFECTIVE_STATUS_SQL`, `DAYS_LEFT_SQL`, `get_membership_counts`, `get_effective_status`) is untouched and stays SQL/SQLite-backed, since `routes/dashboard.py`, `routes/membership_distribution.py`, `routes/notification.py`, and `routes/student.py` all still call them directly against the SQLite mirror. `renew()` additionally now captures which row(s) it's about to flip from `Active` to `Expired` *before* doing so (on both databases), so a failure partway through the request — an `APIError` on the Supabase insert, or a `sqlite3.Error` on the mirror-write — can restore exactly those rows to `Active` and delete the half-inserted new row, rather than leaving a partial expire-without-replace state; the original SQLite-only version didn't need this because a single connection's transaction already made the expire-then-insert atomic, a guarantee that's gone once the two steps span two separate databases. Routes, templates, validation, pricing rules (Membership Settings auto-fill), and the renew workflow are otherwise unchanged. A new helper, `_sanitize_int()` (same shape as `routes/student.py`'s `_sanitize_date()`), converts `duration_days` to an int or `None` before any Supabase call, since Postgres's `duration_days` column is a strictly-typed `INTEGER` unlike SQLite's, which silently accepted the raw form string.

**Why:** requested as the next incremental-migration slice, explicitly scoped to Membership only (`routes/membership.py`/`database/membership_queries.py`), with Payments and Cashbook explicitly excluded from this step even though ADR-19's own "How to apply" section had named Membership+Payment as "the natural pair, since they already share `database/payment_queries.py`/`database/membership_queries.py`." Investigating the dependency graph the same way ADR-16/18/19 did confirmed `memberships.student_id` is a real SQLite FK and four modules — `routes/payment.py`'s `collect()`, `routes/dashboard.py`, `routes/membership_distribution.py`, and `routes/notification.py` — all run raw `JOIN memberships` directly against SQLite; none of those were touched, per this session's explicit scope, so (as with `students`/`enquiries` before it) the SQLite mirror could not be a one-shot bridge or dropped once Supabase became authoritative for `routes/membership.py` itself — it is genuinely load-bearing for those four files, not transitional scaffolding for `memberships` alone. `get_active_membership()`'s move to Supabase-only reads (rather than continuing to read the SQLite mirror, which stays perfectly in sync with every `create()`/`renew()` write in the same request) was a deliberate choice for architectural clarity: it is private to the module being migrated (confirmed by grep — no other file imports it), so there's no reason for the one function `routes/membership.py` uses to guard its own writes to read a mirror instead of the source of truth it just wrote.

**Consequence — a new, real split-brain this slice could not close (TD-37):** `routes/payment.py`'s `collect()` (explicitly out of scope this session, per "Do not migrate Payments or Cashbook in this step") updates `memberships.paid_amount`/`pending_amount` in **SQLite only**. After a payment is collected against an existing membership, the SQLite mirror's `paid_amount`/`pending_amount` are correct (and everything that reads SQLite — Dashboard, Membership Distribution, Notifications, the Student detail page — sees the update immediately), but Supabase's copy of those same two columns — the one `routes/membership.py`'s `index()` now reads — keeps the pre-payment values until the next time that row is written by `create()`/`renew()` (which a payment collection never triggers). This is structurally the same category of consequence as TD-35 (`admins.password`, ADR-16/17) and TD-36 (`enquiries.status`, ADR-18/19) — a column with two writers that don't agree — surfacing here on `memberships.paid_amount`/`pending_amount` instead. Unlike TD-35/TD-36, this one cannot be closed by making the *unmigrated* side write Supabase directly without that being a de facto Payment migration, which this session's explicit scope ruled out; it can only be closed by migrating `routes/payment.py` itself, the natural next slice.

**Consequence — `register()`'s SQLite mirror-insert bridge (TD-35) is unaffected and still fully required, and the `students`/`enquiries` SQLite mirrors are also unaffected:** `memberships` was never one of `register()`'s 7 FK-dependent tables (that FK runs through `students.admin_id`, not directly from `memberships`), so this slice doesn't touch that bridge at all. It also doesn't close any part of the `students`/`enquiries` SQLite mirrors' remaining consumer list from ADR-18/19 — `routes/membership.py`'s own student lookups moved off SQLite onto Supabase (one fewer consumer of the `students` mirror, noted in [09_DEPENDENCY_MAP.md](09_DEPENDENCY_MAP.md)), but the mirror itself is still required for the other modules ADR-19 listed.

**How to apply:** the next slice should be `routes/payment.py` — migrating it would close TD-37 at the source (the same way ADR-19's Students slice closed TD-36 at the source, by moving the *stale writer* onto Supabase rather than working around it from the other side) and would let the `memberships` SQLite mirror's remaining consumer list shrink to `routes/dashboard.py`/`routes/membership_distribution.py`/`routes/notification.py`. Alternatively, one of the remaining SQLite-only tables (`audit_log`, `library_settings`, `membership_settings`, `backup_log`, `security_settings`) could be picked instead to keep closing `register()`'s FK dependencies — that work is orthogonal to `memberships`/`payments` and can proceed independently.
